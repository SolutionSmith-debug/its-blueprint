## 1. Purpose

Replace the inbox-and-PDF safety-report submission path with a web portal that field PMs use to submit daily safety paperwork directly. Field PMs log in, pick a project, pick a form, fill structured fields, capture per-worker signatures on-screen, and submit. The structured submission is handed to the existing `safety_reports` pipeline via a **send-free Cloudflare Worker queue** that a Mac-side poller (`portal_poll.py`) drains — the **Python PULL transport** (no email shim); the pipeline renders the matching PDF in Python (Option B) as a derived artifact.

The shift is from "extract structure from handwritten PDFs after the fact" to "enforce structure at entry, render the PDF as a derived artifact." This eliminates OCR confidence loss, handwriting illegibility, multi-day packet splitting, Spanish-name normalization fragility, and most of the SDK-vs-Live extraction surface a prior `jha_split.py` plan would have built.

**The portal facilitates; the Python scripts run the pipeline.** The portal is a thin capture-and-hand-off surface. It owns authentication, form display, signature capture, and a durable receipt of each submission. It does **not** run the pipeline: it never touches Smartsheet or Box, never calls Anthropic, and never sends a customer-facing message. The per-submission PDF render, all Smartsheet writes, all Box filing, weekly compilation, and the gated weekly send are owned by the existing Python pipeline ([`safety_reports`](../safety-reports/mission.md), which holds the only write credentials). This division is the [External Send Gate](#7-foundation-invariants-inherited) two-process model expressed as a deployment boundary, not just a code boundary.

## 2. Audience

Four distinct audiences with different surfaces:

- **Field PMs (portal users):** Daniel, Devin, Henry, and equivalent field-side roles on each active project. They submit forms. They never see Smartsheet, Box, or any ITS operator surface.
- **Admins (in-browser dashboard — CEO / head PM, e.g. `stephens.jacob` / `finkhousen.ben`):** A **role-gated admin SPA** — distinct from the operator-CLI of Phase 7 — for **account management** (create / edit-login / change-role / disable / delete, with an atomic last-admin guard) and **form management** (the Form Editor + the publish pipeline + a live status monitor; §14). They can also **submit-as** ("filled out as") — enter a form on a field PM's behalf with dual attribution (the canonical HMAC payload + downstream pipeline are byte-unchanged). Admin sessions carry a **30-minute sliding idle window** (vs the field PM's 90-day session), with a bounded keep-alive so an open dirty editor isn't bounced mid-edit (§13). This is a *browser* audience, separate from the operator-maintainer's CLI/Smartsheet surfaces below.
- **Office PMs (administration users):** Whoever maintains the active-jobs list. They edit `ITS_Active_Jobs` (job list, contacts, Active/Inactive/Archived status) in Smartsheet. They never log into the portal as a field user.
- **Approvers + operator-maintainer (initially Seth, eventually a trained Evergreen employee):** Approvers review, edit, and approve the compiled weekly report in the `WSR_human_review` sheet. The operator provisions portal accounts (now also possible in-browser by an admin, above), monitors submissions via existing operator surfaces (`ITS_Daemon_Health`, `ITS_Errors`), and ships new forms — a commitment that **now has a mechanical expression**: the in-browser Form Editor + publish pipeline (§14) replaces hand-editing a declarative definition + catalog row, while the validators / renderers / send gate still require a developer PR.

Submissions flow into the downstream surfaces the Python pipeline owns: per-job **week sheets** in the `ITS — Safety Portal` workspace, the Box job/week folder tree (system of record), the weekly compiled PDF, and the customer-facing approval sheet `WSR_human_review` (see [Integration](#10-integration-with-safetyreports-workstream)). The portal writes to **none** of these directly.

## 3. Inputs

- **Field PM credentials** — `lastname.firstname` username + bcrypt-hashed password (per-user, operator-provisioned via admin route). Session cookie persists 90 days.
- **PM-selected work date** — ISO date, unbounded backdating window per operator decision (risk-accepted; see [Risks](#12-risks)). The work date determines the **Saturday→Friday week** the submission buckets into (weekend work rolls forward to the following Friday).
- **PM-selected project** — from the portal's own D1 store, populated from Smartsheet `ITS_Active_Jobs` out of band. **The portal never reads Smartsheet at request time.** Only `Active` jobs are offered. The job's permanent key is its **6-digit `AUTO_NUMBER` Job ID** (`JOB-000001`); the human-readable **Project Name** is display only.
- **PM-selected form** — from the catalog the portal serves out of D1 (populated from `ITS_Forms_Catalog`), filtered to the selected job, with parent/variant collapse under a third picklist (see [Forms](#9-decisions-locked)).
- **Form payload** — per-form structured fields, schema-validated client-side and server-side. Worker roster entries with per-row name + company + signature (SVG path data, optional). Equipment checklists are **tri-state (OK / NOT OK / N/A)**.
- **Pipeline receipt callback** — the Python pipeline POSTs back to the portal after the submission is filed, flipping the D1 receipt so the portal can show "received & filed." The callback is **fail-closed**: an unconfirmed submission stays visibly pending.

## 4. Outputs

- **HMAC-signed D1 queue entry** — on `POST /api/submit` the send-free Worker HMAC-signs the submission and stores it in D1 as a durable queue entry. The Mac-side `portal_poll.py` daemon pulls it over the bearer-gated `GET /api/internal/pending`, verifies the HMAC, and hands it to the pipeline. **No email is sent.** Dedup keys on the submission UUID.
- **D1 submission record** — the portal's durable record of every submission (payload, work date, PM, HMAC, `box_verified` / `filed_at` receipt fields). This is the portal's only native system of record; everything customer-facing lives downstream.
- **(Downstream, pipeline-owned, listed for completeness)** — a per-submission PDF rendered in Python (Option B), filed to Box and reflected as a row in the per-job **week sheet**; the weekly **compiled PDF**; the `WSR_human_review` approval row. **Box is the system of record**, its folder tree mirroring the Smartsheet job/week hierarchy.

## 5. Success criteria

- **Zero handwritten-PDF safety submissions from onboarded field PMs by 60 days post-validation cutover.** Measured via week-sheet `source` (portal vs legacy).
- **OCR-derived extraction surface retired for safety reports.** A `jha_split.py`-style extraction plan is superseded; structured payload at entry replaces extraction at receipt.
- **PM submission median latency < 2 minutes** from login to confirmation, including form fill + signature capture.
- **Zero customer-facing send caused by AI-process compromise.** Guaranteed structurally: the portal cannot send customer-facing mail, and the only external send (`weekly_send`) reads an approved, human-edited row from `WSR_human_review` (Invariant 1, two-process).
- **A trained Evergreen operator can ship a new form (or variant) end-to-end** — declarative definition + `ITS_Forms_Catalog` row — in under 1 hour with AI assistance, with the TS display runtime and the Python renderer both honoring the single definition without drift.

## 6. Out of scope

- **No payment, timesheet, or financial data.** Portal stores form payloads + signatures only. Per the Q2 threat model, this scoping is the substantive defense behind the bcrypt-hashing decision — the portal stores nothing of direct value.
- **No personnel master database in v1.** Worker names are free-text per row. Name normalization is deduplicated only within the weekly aggregator's case-insensitive walk.
- **No mobile native app.** Browser-based; signature canvas uses standard touch events.
- **No idle session timeout.** Explicit Sign Out only (zero-friction operator goal).
- **No self-service password reset for field PMs.** Operator-mediated reset only.
- **No retroactive customer send for backdated submissions.** A sent weekly report is frozen at send time; the operator decides whether to send an addendum (manual).
- **No AI calls from the portal at submission time.** The portal is structured-payload-only; the downstream `weekly_generate` step (owned by `safety_reports`) is the only AI call site in the pipeline.

## 7. Foundation Invariants Inherited

Per [Foundation Mission v11](../../doctrine/foundation-mission.md). The Safety Portal touches **both** external-bound send (the weekly report) **and** external-content ingestion (every field-PM submission), so it restates **both** invariants verbatim from doctrine, then states the portal-specific application. The canonical statements below are quoted from doctrine; the doctrine wins on any conflict.

### Invariant 1 — External Send Gate

Canonical statement ([Foundation Mission §Invariant 1](../../doctrine/foundation-mission.md#invariant-1-external-send-gate-unchanged-from-v7)):

> No external transmission without explicit human approval. Permanent, not time-bounded. Earlier framing in Op Stds v4 that described review as a 30-60 day window is superseded.
>
> - Every workstream that produces customer-facing output uses a `<Workstream>_Pending_Review` Smartsheet with Approved for Send / Approved By / Approved At / Sent At / Send Status columns.
> - Two-process model. Generation scripts (which call the Anthropic API) have zero send capability. Send scripts (which transmit) have zero AI step.
> - Successful prompt injection at the AI layer cannot cause external transmission, because the AI is in a different process from the transmitter.
> - Enforced at the code level by `tests/test_capability_gating.py` — every generation script and every send script is registered to the appropriate list there.

**Portal application.** The portal produces no customer-facing output and has **no transmission capability at all** — the Cloudflare Worker is **send-free** (no Graph/Resend, no email; verified in `worker/index.ts`). Submissions are stored in D1 and **pulled** by the Mac-side `portal_poll.py`; the portal never pushes anything outward. The customer-facing artifact (the weekly report) is gated by the `WSR_human_review` sheet, which **is** the workstream's `<Workstream>_Pending_Review` surface: the **editable email body in that row is the source of truth for the send**, and `weekly_send` (no AI step) transmits only after a human sets `Approve for Scheduled Send` / `Send Now`. The two transmission-relevant components both live on the Python side, **inside** the AST capability-gate (`tests/test_capability_gating.py`): `portal_poll.py` (a puller — send-free) and `weekly_send` (the only customer-facing sender, in `SEND_SCRIPTS`). Choosing a Python pull over a TS email shim *closed* the trust-boundary gap a shim would have opened — a TS sender transmits outside the Python AST scan (decision `decision_phase5-portal-transport`).

**Two-mode send transport (PR #275 `13ef2bc`, v5) — the transport changed, the gate did not.** `weekly_send` now selects its Graph transport by compiled-packet size: **inline** when ≤ 2.5 MB (`UPLOAD_SESSION_THRESHOLD_BYTES` = 2,621,440 bytes, owned by the caller; switch on strict `>` so exactly 2.5 MB still goes inline via `graph_client.send_mail`), a **Graph upload-session** above that (`graph_client.send_mail_large_attachment`: draft → `createUploadSession` → chunked PUT honoring `nextExpectedRanges`, `UPLOAD_CHUNK_SIZE` 3,276,800 B → send), and **HELD** beyond Graph's ~150 MB ceiling (`UPLOAD_SESSION_MAX_BYTES` = 157,286,400) — `Send Status = held_oversized_packet`, **operator-actionable, never silently dropped**. The oversized-packet refusal is evaluated **before** the write-ahead `SENDING` marker; the inline-vs-upload-session branch is selected **after** it. None of this touches the gate: `weekly_send` remains the only customer-facing sender, AI-free (in `SEND_SCRIPTS`, `anthropic` AST-forbidden), human-approved on `WSR_human_review`, recipients resolved at send time from `ITS_Active_Jobs`. The 2.5 MB threshold is a deliberate heuristic (not empirically measured against the live Graph tenant — too-low only routes a sendable packet down the slightly slower chunked path); §43 runbook `docs/runbooks/safety_photo_path.md`, ADR-0001.

### Invariant 2 — Adversarial Input Handling

Canonical statement ([Foundation Mission §Invariant 2](../../doctrine/foundation-mission.md#invariant-2-adversarial-input-handling-revised-in-v8)):

> All content originating outside the operating customer tenant is untrusted data. Six-layer defense (v8 expanded from v7's five). The invariant itself is unchanged from v7; Layers 1 and 6 are revised/added (v8), and Layer 5 is recharacterized as a detection tripwire, not a defense layer (v9, see below).

Portal application of the six layers (every field, every signature, every PM-entered name is untrusted data):

- **Layer 1 (Authentication boundary + scope).** Portal users authenticate via username + bcrypt-hashed password (cost factor 10) against D1. No self-registration; accounts are operator-provisioned. This is the portal's equivalent of the `ITS_Trusted_Contacts` allowlist for the email-intake path. On the transport side, the Layer-1 controls are the **bearer token** gating `/api/internal/*` and the **per-submission HMAC** the Mac-side poller verifies (`shared/portal_hmac.py`) before filing — there is no inbound mailbox to allowlist (the email shim is retired).
- **Layer 2 (Untrusted-content tagging).** The portal calls no Anthropic API. Downstream `weekly_generate` wraps any PM-entered text in `<untrusted_content>` tags via `shared/untrusted_content.py` before any model call.
- **Layer 3 (Capability gating).** The portal has no Smartsheet write, no Box write, no Anthropic, and no customer-facing send. Enforced by Cloudflare Workers binding scope and by `tests/test_capability_gating.py` on the Python side.
- **Layer 4 (Structured output enforcement).** Submissions conform to per-form definitions; payloads failing schema validation are rejected (client + server) and never enter the D1 queue. The Python side validates again before any write (defense in depth).
- **Layer 5 (Anomaly logging — tripwire, not a defense layer).** Per v9 framing, this is a low-effort detection tripwire, not a barrier. Suspicious patterns in PM-entered text route the submission to `ITS_Review_Queue` for human attention; it is never relied on for prevention.
- **Layer 6 (Attachment screening) — now an exercised, image-constrained attachment class (v5; was "N/A — no attachments").** Signatures remain SVG path data (vector, not raster). As of PRs #271/#272 the portal **does** accept attachments, but only a **constrained image class**: a header-level `photo` input, JPEG/PNG only, bounds-gated at the send-free Worker, and **§34-screened in code** on the Mac before any Box upload or PDF render. The [Operational Standards §34](../../doctrine/operational-standards.md) pipeline is realized as `safety_reports/photo_screen.py` — Layer 1 magic-byte check → Layer 2 Pillow `verify()` + decompression-bomb cap + forced re-encode (which strips all metadata) → Layer 3 ClamAV-on-raw-bytes (config-gated, default off); a non-clean disposition **refuses the whole submission before filing** — MALICIOUS fires a CRITICAL naming the account + a security-flagged Review-Queue row, SUSPICIOUS refuses identically at WARN. Full as-built in **§15**. (PMs still cannot attach arbitrary files — no PDFs/Office/executables; the input is image-only.)

**HMAC-verified portal payload (load-bearing).** Beyond the six layers, the send-free Worker HMAC-signs every submission (HMAC-SHA256 of the canonical payload) before storing it in D1; the Mac-side `portal_poll.py` re-verifies it (`shared/portal_hmac.py`, constant-time `compare_digest`) before handing the submission to `intake` — a mismatch is rejected/flagged and never filed. The `HMAC_SECRET` lives in a Cloudflare Workers Secret with a macOS Keychain mirror for the poller. **Photos did not change the canonical payload (v5):** the canonical string is unchanged — `uuid \n job_id \n form_code \n work_date \n payload_json` — and base64 photos ride *inside* `payload_json`, so the existing HMAC covers them with **zero signing changes**. A regression test (`safety_portal/test/photos.test.ts`) recomputes the canonical HMAC over a photo-bearing payload byte-for-byte and locks this.

## 8. Self-containment and workspace-as-approval-authority

The whole safety workflow lives in **one standalone Smartsheet workspace, `ITS — Safety Portal`**. The `Safety Portal` folder (`ITS_Active_Jobs` + `ITS_Forms_Catalog`) was **moved** there from `ITS — Operations` with **IDs preserved**; the per-job week sheets and the `WSR_human_review` approval sheet live in the same workspace. Shared infrastructure config (`ITS_Config`, `ITS_Errors`, `ITS_Trusted_Contacts`, `ITS_Review_Queue`, `ITS_Quarantine`) stays in `ITS — System` and is read by sheet ID — it is shared infra, not safety-specific.

Two consequences are load-bearing doctrine:

- **Workspace access = approval authority.** Sharing the `ITS — Safety Portal` workspace is the send-gate access control: the people who can review/edit/approve `WSR_human_review` are exactly the workspace's shared collaborators. There is no separate per-sheet ACL to maintain.
- **The system ships independent of the Forefront/demo structures.** Safety-path code must not depend on non-safety sheets. The Safety Portal can be stood up, demoed, and handed over without the portfolio/operations/archive workspaces. This is why it is a *standalone* workspace rather than a folder inside the five-workspace audience-separated topology of [Operational Standards §23](../../doctrine/operational-standards.md). **Doctrine reconciliation — done:** [Operational Standards v17](../../doctrine/operational-standards.md) (2026-06-05) now acknowledges this sixth, standalone, approval-gated workspace in §23/§24 — doctrine and the as-built are aligned. (Resolves the flag raised in the [2026-06-05 blueprint-update session log](../../session-logs/2026-06-05_safety-portal-blueprint-update.md); the v17 bump was applied in the [transport+clean-break session](../../session-logs/2026-06-05_safety-portal-transport-cleanbreak.md).)

### 8.1 F22 approval authority is realized as workspace membership (PR-E #183)

As of PR-E (#183, exec `595a469`) the v2 principle above is **realized in code** — "sharing IS the gate" is no longer aspirational. The F22 send-gate predicate (`shared/approval_verification.py` `verify_approval` — match the `Approved for Send` cell-history modifier's email against the authorized set) is **unchanged**; what moved is the **source of the authorized set**:

- The authorized-approver set is now resolved **live** from `ITS — Safety Portal` workspace membership — `smartsheet_client.list_workspace_share_emails(WORKSPACE_SAFETY_PORTAL)` reads `GET /workspaces/{id}/shares?includeAll=true` and returns the lowercased emails of every **USER** share (any access level), called by `weekly_send_poll._load_authorized_approvers`.
- The former **`safety_reports.authorized_approvers` ITS_Config allowlist is retired** (seed removed 2026-06-06; no live code reads such a row). The mechanism is gone; only the *naming* survives (`_load_authorized_approvers`, which now resolves from shares), tombstone comments, and historical session logs.
- **Fail-closed, never fail-open:** an empty resolved set → `EMPTY_ALLOWLIST` → all sends blocked.

Two edge cases are load-bearing doctrine and must be recorded honestly:

- **Group-share expansion is a known pre-prod gap.** Only USER shares carry an email; **GROUP shares are excluded** (no email field), so a workspace shared *only* to a Smartsheet group resolves to the empty set → all sends blocked. Mitigation today: share with **individuals**; group-membership expansion is a documented follow-up (helper docstring + cutover checklist + a unit test all record it).
- **Workspace-owner inclusion is *not* handled in code.** The resolver injects no owner and filters no access level — whether the workspace **owner** appears in the approver set is an **unstated dependency on the Smartsheet `/shares` response shape**. This is an open question, not a guarantee; do not assert the owner is covered.

## 13. Admin dashboard and account model (as-built)

The in-browser **admin dashboard** (§2) is the second major surface the portal grew in this window — a role-gated SPA for the CEO / head PM, architecturally distinct from the operator-CLI of Phase 7. Its security model is **server-authoritative**: the SPA only chooses what to *render*; every admin action is re-gated server-side (Invariant 2 — never trust the client).

- **Role model.** Migration **0007** adds `users.role` (`TEXT`, `CHECK (role IN ('submitter','admin'))`) plus an append-only **`audit_log`**; **0008** adds submit-as attribution (`actor_username` / `submitted_as`). Account mutations and their audit rows are written in **one atomic D1 batch**.
- **Per-request D1 authority (fail-closed).** `requireSession` reads `disabled` + `role` + `session_epoch` in a **single** per-request `SELECT` keyed on the username; **any** D1 error → `401` (a DB blip neither grants access nor crashes the request). Role is authoritative from **D1, not the cookie**, so a demotion or disable is effective immediately; `coerceRole` fails safe (unknown / null → `submitter`, never `admin`).
- **Session-epoch revocation (migration 0009).** Logout and password-change `UPDATE users SET session_epoch = session_epoch + 1`; an outstanding cookie's epoch snapshot falls behind the column and is rejected at `requireSession`. This is the captured-cookie kill switch (a pre-epoch cookie carries `epoch 0` = the column default, so existing sessions survive).
- **Atomic last-admin guard.** Demote / disable / delete evaluate the guard **in-WHERE** — `AND (SELECT COUNT(*) FROM users WHERE role='admin' AND disabled=0) > 1` is injected into the UPDATE, so the "would this leave zero admins?" check and the write are one statement (no TOCTOU window — it survived a live race in the security audit, §G29).
- **Operator-CLI break-glass.** The bearer-gated `/api/internal/admin/*` routes (the `portal_admin.py` operator CLI, gated by the **separate** `PORTAL_ADMIN_API_TOKEN` — privilege separation from the poller/publish token) are **deliberately NOT last-admin-guarded**: the CLI is the recovery path *out* of an accidental zero-admin state.
- **Admin idle window.** Admin sessions carry a **30-minute sliding** server-side window (re-issued on each active request) vs the field PM's 90-day session — widened from 5 min in #258 (§G34). The SPA pings `/api/session` on activity to slide the window and logs out proactively at idle; while a **dirty form-editor draft is open** it adds a bounded wall-clock keep-alive so a briefly-backgrounded tab isn't bounced — but proactive logout still fires in *both* modes, so an *abandoned* dirty tab still dies at ≈30 min (the unbounded-session hole the adversarial review caught; `draftCache` preserves the work regardless). The **server cookie window is the real boundary**; the SPA keep-alive is UX.

## 14. Form management and publish pipeline (as-built)

This is the doctrine-significant addition of the window: the form-editor publish pipeline is a **second privileged-actuation gate**, architecturally parallel to the [External Send Gate](#7-foundation-invariants-inherited) — the two-process model applied to **code actuation** rather than external send.

- **The editor.** A **B8 sectioned form builder** (`FormEditor.tsx` — field-type palette, section headers, explicit field rows), chosen over a YAML textarea (too error-prone for the Successor-Operator) and over drag-and-drop (no structural benefit — the form vocabulary is closed). A **live `FormRenderer` preview** sits beside the builder (render-parity confidence in-tab), and a **per-account localStorage draft cache** survives the admin idle logout / reload (cleared only on Discard or a successful publish).
- **The publish state machine.** `publish_requests` (migration **0010**) carries the status vocabulary **`queued → validated → tested → merged → live → archived | failed`** (the CHECK constraint is authoritative). The SPA **Status Monitor** renders it with operation-aware stepper labels (Publish ends *Live · Archived*; Retire ends *Removed · Done*).
- **Three-layer validation (fail as early as possible).** The editor warns client-side on a parent/variant-grouping mismatch; the Worker's `publishValidation.ts` rejects a malformed enqueue with `400`; the daemon's `apply_publish` (the pure manifest transform in `safety_reports/publish_manifest.py`) re-validates against **live git HEAD** before any git op. The variant-mixing guard lives at all three layers, not daemon-only.
- **The daemon is the sole privileged actuator (the code-actuation gate).** The cloud Worker is **send-free AND code-free**: it validates the payload, enqueues to `publish_requests`, and exposes queue-management endpoints — it **never** touches git, the filesystem, or a Cloudflare deploy. The Mac **`publish_daemon.py`** is the *only* process that can commit/merge/deploy; its `_actuate` orchestrator runs per request: **claim → re-validate vs git HEAD → commit** the new/updated form file (**append-only**, design **C1**) on a per-request branch + open a PR → **synchronously wait for the CI render-smoke gate** (`_wait_for_ci` polls `mergeStateStatus == CLEAN`; `CI_TIMEOUT_S = 900`) → **squash-merge + verify the merge-commit OID** → **local `wrangler` deploy** → post-deploy liveness ping → **Box blank-archive regeneration** (run with `sys.executable`, never bare `python` — the #241 launchd-PATH fix) → stamp the monitor. Per-request deploy is serialized per parent (design **C8**); a failed cycle is self-healed by `_unstrand_if_needed` (#224). A compromised Worker therefore **cannot publish arbitrary form definitions or deploy to the live portal** — exactly mirroring how `weekly_send.py` is the only process with Graph send capability. The gate is verified at the import level by `tests/test_capability_gating.py` (the daemon is in the generation list; no send capability).
- **C12 = A clearance (fully-automatic publish).** Because a form publish commits + deploys *code*, fully-automatic publish (no human merge checkpoint) is the **highest capability class** — and is a deliberate **per-category clearance** (the first ITS per-category code-deploy clearance, `decision_phase2-form-editor`), load-bearing on the mandate that the automated gates (CI 3-renderer render-smoke + server/daemon validation + **branch-protection required checks**, which the daemon's merge-on-CLEAN makes load-bearing) **fail closed and STOP a bad publish**, and that the portal **detects progress + ALERTS on any stall — no silent stall**. The forensic audit reviewed this auto-merge-and-deploy posture (finding H1) and **accepted it as by-design** under C12=A.
- **Legal-invariants-in-code — now realized as a required-content manifest (v5 upgrade from ◐).** Structural validation (closed vocabulary + reserved-key denylist + cross-section-unique keys + hard bounds) and the CI render-smoke gate were already live ✔; the **per-identity required-content manifest** (the JHA "if conditions change…" footer / equipment lock-tag-out line enforced as *required content* on every version, not merely present in a seed) is now **shipped as `safety_portal/required-content.json`** and enforced at **both** C3 layers — the Worker enqueue gate (`publishValidation.ts validateDefinition`) and the daemon's authoritative re-check vs live HEAD (`safety_reports/publish_manifest.apply_publish`). The manifest carries strict `parents` entries for the legal forms — `jha` (the "REVIEW AND REVISE THE PLAN" footer), `equipment-preinspection` (the "ALWAYS lock/tag-out unsafe equipment" line), `toolbox-talk`, `incident-report`, and (v5) **`lifting-plan`** — while `defaults_for_new_identities` (`required_signature_inputs_min: 1`) keeps the demonstrated operator create-new-type flow working. **A strict entry is a legal-floor (doctrine-adjacent) change**: the file states it verbatim — *"CHANGING A STRICT ENTRY IS A LEGAL-FLOOR (doctrine-adjacent) CHANGE: high-capability-class, Tier-3/Seth only — the Successor-Operator must NOT edit this file"* (runbook `docs/runbooks/safety_portal_forms.md`). The `lifting-plan` strict entry (50,000-lb threshold, the >75%-of-rated-capacity critical-lift line, 3-year retention, sign-before-lift, and the two required field keys `pct_rated_capacity_used` + `lift_director_name`; see §15) is **ratified in this v5 pass** under that rule.
- **Failed-publish recovery.** A failure (commit through deploy) retains the operator-composed `definition_json` on the queue row; the Status Monitor exposes **"Edit & re-publish,"** re-opening the editor pre-populated — a CI timeout or Box error never discards a multi-field definition.

**Realized form-maintenance commitments (absorbs the chat-side `safety-portal-form-maintenance.md`, now retired):** forms-as-version-controlled-config ✔ — now **operator-publishable end-to-end**, demonstrated by the incident-report create→v2→v3 chain (reqs 16–18); preview-before-live ✔ (the FormEditor live renderer); invariant-validators ◐ (structural ✔, legal-content manifest pending — pointer above). The **operator-editable vs build-frozen dividing line now has a mechanical expression**: catalog/forms changes flow through the operator publish path, while validator / manifest / renderer / send-gate changes require a developer PR.

## 15. Site photos and §34 image-class screening (as-built — v5)

The first attachment surface the portal grew. v4's §7 Layer-6 conditional ("if the portal ever gains file-attachment capability… §34 applies") is now **exercised in code** — and deliberately **constrained to an image class** so the §34 surface stays small. PRs #271 (`fadd53f`, Worker + SPA capture) and #272 (`5a979e2`, Mac-side screening + embed + Box originals). The same pattern is the canonical instantiation proposed propose-only for Op Stds §34 (§Doctrine flags).

- **Transport — D1-inline, no R2.** Photos ride **D1-inline** base64 within the per-submission budget; `worker/types.ts` keeps its "No R2" stance (removed 2026-06-05). **Box remains the system of record.** R2 (D1 carries only an object key; the Mac fetches bytes at screen time) is the **recorded upgrade path** for when field crews need > 4 full-res photos or the budget is raised — deferred in **ADR-0001** (`docs/adr/0001-portal-photo-transport-d1-vs-r2.md`).
- **Input type + Worker bounds gate (send-free).** A `photo` input in `meta-schema.json` (input enum), **header-level only** (`publishValidation.ts` rejects `photo` as a table column; `max_count` 1..4). The Worker enforces, before storing: **JPEG/PNG magic** (`FF D8 FF` / `89 50 4E 47`, `photoMagicOk`); **≤ 400 KB decoded/photo** (`PHOTO_MAX_BYTES = 400_000`); **≤ 4/field** (`PHOTO_MAX_PER_FIELD`); **≤ 8/submission** (`PHOTO_MAX_PER_SUBMISSION`); and a request **payload cap raised 1.0 → 1.8 MB** (`PAYLOAD_MAX = 1_800_000`).
- **Canonical HMAC unchanged (regression-locked).** Photos ride inside `payload_json`; the canonical string (`uuid \n job_id \n form_code \n work_date \n payload_json`) and the HMAC are **byte-unchanged** — zero signing changes. Locked by `safety_portal/test/photos.test.ts` ("photos ride payload_json under the UNCHANGED canonical HMAC").
- **Client capture + caption-then-strip.** `PhotoField.tsx` uses `accept="image/*" multiple` with **no `capture` attribute**, so field crews get the native camera/gallery chooser. **EXIF orientation is baked in at re-encode** (`createImageBitmap(..., {imageOrientation:'from-image'})`), and `taken_at` / GPS are read to a sidecar (`exif.ts`) to print as a PDF caption — then the client canvas re-encode drops the EXIF (the first "strip"). This is **caption-then-strip**: keep the provenance words, destroy the metadata bytes.
- **Mac `photo_screen.py` = §34 Layers 1–3.** Before any render or Box upload, `safety_reports/photo_screen.py` runs **Layer 1** magic + size; **Layer 2** Pillow `verify()` + a **decompression-bomb cap** (`MAX_IMAGE_PIXELS = 24_000_000`) + a **forced JPEG re-encode** (rebuilds via `Image.frombytes`, so all `.info` EXIF/ICC/XMP/COM is gone — the second, authoritative strip); **Layer 3** ClamAV (`_clamav_scan`) **on the RAW original bytes** (a re-encode would strip a payload before clamd saw it), gated by `safety_reports.photo_screen.clamav_enabled` (`CFG_PHOTO_CLAMAV`), **default OFF**. Disposition vocabulary is `clean | suspicious | malicious`.
- **Disposition handling (fail-toward-not-filing).** A non-clean photo **refuses the whole submission** (`screen_photo` → `_portal_photo_refusal`, routed to the Review Queue as a drain-eligible `ProcessResult`, **before** any render / Box / Smartsheet write). **MALICIOUS** → a `Severity.CRITICAL` page (the error-log fires *first*, naming the offending account — "DISABLE portal account `{actor}`") + a `review_queue.add(..., security_flag=True)` row. **SUSPICIOUS** → the **same refusal**, at `Severity.WARN` (not CRITICAL) and **without** the disable-account directive. Both dispositions are terminal refusals to the Review Queue — the difference is the severity + the account-disable page, **not** the filing outcome (neither files).
- **Box originals + renderer consumes only screened bytes.** Sanitized originals (the §34-screened re-encoded JPEGs, `screened_photos` — never the raw upload) file to **`ITS Photos/<submission_uuid>/`** under the job folder (`_file_portal_photos`). The PDF renderer reads **only** `submission['screened_photos']` (`form_pdf.render_submission_pdf`); the header section skips `photo` fields and a stray table-column photo renders `[photo omitted]` — the renderer **never touches raw upload bytes**.
- **Follow-on (#279).** The Form Editor's `editorValidation` was fixed to accept the `photo` input (derive `INPUTS` from `FIELD_INPUTS`, with parity tests), so a `photo` field is now authorable through the publish pipeline (not just hand-written); a `photo-test-v1` form (#281, req 19; parent `test`) exercised it end-to-end.

## 16. Request-driven PDF download and in-portal Form Request (as-built — v5)

A document-retrieval surface, send-free and authenticated. PR #274 (`814aec6`) added the submitter receipt + canonical-PDF cache; PR #276 (`213d076`, migration 0012) generalized it to an any-authenticated-account Form Request with **requester-bound** downloads; PR #280 (`ff00308`) added month-year + form-type filters to the browse.

- **Canonical-only, no fourth renderer.** `GET /api/submissions/:uuid/pdf` reassembles the **Box-filed PDF** from a chunked D1 cache and streams it (`Content-Disposition: attachment`) — **byte-identical to the filed copy, no browser render**. PR-4 deliberately added no renderer (render-parity preserved): the existing Python `form_pdf` output filed to Box is the single source.
- **Chunked D1 cache, 24 h TTL.** Migration `0011_add_pdf_request_cache.sql` adds `filed_pdfs(submission_uuid, chunk_index, chunk_total, chunk_b64, created_at)` — chunked because a photo-bearing PDF + base64 inflation exceeds the ~2 MB D1 row ceiling (`MAX_CHUNKS = 8`, `CHUNK_DECODED_MAX = 1_000_000` ≤ 1 MB decoded/chunk). The cache expires at **24 h** (`PDF_CACHE_TTL_S = 86_400`).
- **Post-back lives in `portal_poll`, never blocks filing.** The best-effort Box-download → chunk → `upload_filed_pdf` post-back is performed by **`portal_poll._service_pdf_requests`** (calls `shared/portal_client.py`), **not** `intake.py` (intake makes zero `portal_client` calls — capability/preservation). The whole pass is fenced (`try/except` → `Severity.WARN`, `error_code=portal_pdf_service_failed`) and per-item fenced: a post-back failure **never blocks the intake drain**. `box_file_id` is recorded at mark-filed time (`POST /api/internal/mark-filed`).
- **Home-screen Form Request — any authenticated account may browse + request.** `HomePage.tsx` adds a **Form Request** card → `FormRequestPage`. `GET /api/filed` is `requireSession`-gated only (any authenticated account browses — mirroring the submit model), and **404s unless the job is active** (`SELECT 1 FROM jobs WHERE job_id=? AND active=1`). The job list is `api.fetchJobs` → `GET /api/jobs` (`jobs WHERE active=1`) — **the same Active-Jobs full-replace sync** (`POST /api/internal/sync`) the fill flow uses; one shared D1 `jobs` table.
- **Requester-bound downloads (migration 0012).** `pdf_requests(submission_uuid TEXT, account TEXT, requested_at INTEGER unixepoch, ready_at INTEGER, PRIMARY KEY (submission_uuid, account))` **supersedes** PR-4's per-submission `pdf_requested` flag (the Part-A submitter flow becomes the first row in this table, behavior preserved). A grant is valid **24 h from request** (`requested_at > unixepoch()-86400`), **re-requestable** (`ON CONFLICT(submission_uuid, account) DO UPDATE SET requested_at=unixepoch(), ready_at=NULL`); a different account → **404, no enumeration**.
- **Two-stage prune + telemetry.** `prune.ts`: **Stage 1** at 90 d (`SUBMISSION_RETENTION_DAYS`) strips `payload_json=''` while keeping the metadata row incl. `box_file_id` (browse is submission-level); **Stage 2** deletes 30 d (`INACTIVE_JOB_GRACE_DAYS`) after the job goes inactive; cache chunks are swept when no live request (`DELETE FROM filed_pdfs WHERE submission_uuid NOT IN (SELECT submission_uuid FROM pdf_requests)`); an **unfiled row (`box_verified=0`) is never evicted** (the Mac always gets one chance to file). The prune emits a **`dbSizeBytes`** telemetry value (scheduled handler; `DB_SIZE_WARN_BYTES = 6_000_000_000`).
- **Why in-portal and not email (owner decision 2026-06-12).** Delivery is **in-portal only**. An email-delivery variant (account-registered recipients + auto-send of filed PDFs) was **declined**: the Worker is **send-free by Invariant 1** (no email path exists in it at all), so in-portal delivery is the architectural default — and an auto-send Form Request would have been **ITS's first automated external send**, an External-Send-Gate matter. The only documented delivery mechanism decision is **requester-bound 24 h binding at the D1 layer** (vs a session-cookie/KV token). *Honesty note: this owner decision lives in chat + this mission record; there is no separate ADR/exec artifact for the declined email variant (the brief-validator pass confirmed none exists — the absence is consistent with the send-free posture). Any revival is **Seth-only, high-capability category 1.***
